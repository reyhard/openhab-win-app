### Current mounted-Location follow-up

At `b1fd1660ae34bf8883dac24be3e7ef1edd251975` (`fix: validate mounted sitemap event locations`), focused RED first proved that Core accepted malformed `/wrong/sitemaps/events/{id}` locations and that the probe rejected a mounted `/openhab/rest/sitemaps/events/{id}` location. Focused GREEN passed `11/11`; the complete Core suite then passed `146/146` in 959 ms, and the controlled loopback probe integration passed in approximately 45 seconds.

The follow-up makes both validators require `rest/sitemaps/events/{nonblank-id}` as the final four path segments, allowing an optional mount prefix and preserving absolute/relative plus query/fragment cases. A present HTTP `Location` header remains authoritative: a malformed or blank header causes subscription validation to fail rather than silently falling back to a legacy body location. This is the existing Core and fake-probe header-precedence contract. The fake server now proves a mounted `/openhab` base path and mounted `Location` route end-to-end.
